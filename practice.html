Digital Accessibility Specialist

Tasks

  • Find and fix barriers in websites
  • Contribute to ongoing accessibility processes
    • Improving workflows or policies
  • Build sustainable practices
    • So accessibility is always considered
  • Develop documentation, guidelines, and other resources
    • i.e., checklists, short guides on key accessibility tasks
    • Record quick tutorials or live demos for remote settings
    • So that others follow best practices long-term
    • This ensures knowledge transfer after contract
  • Provide direct consultation & support:
    • Host Q & A sessions to clarify doubts (screen sharing, digital resources)
    • Answer questions from faculty, developers, or content creators
  • Train faculty & staff
    • Collaborate with IT, faculty, & content creators
      • To prevent, not fix later
  • Review of the following (but not limited to):
    • Keyboard navigation
    • Color Contrast
    • Image alt text (even in PDFs!)
    • Logical tab focus order (i.e., HTML semantics, even on apps!)
      • Don't forget to check for interactive elements
      • Proper reading order
      • Correct heading structure
    • Tagged elements (i.e., PDFs)
    • Labeled form fields (fix missing labels on form fields)
    • Focus states (e.g., outline)
    • Aria-Roles (supplements semantic HTML's limitations)
      • Ensure their usage is appropriate
    • Error messages (ensure these are announced by assistive tech)
    • Acrobat forms
  • Correct application of ARIA attributes:
    • There are roughly 50-60 core ARIA roles
    • Breakdown: ARIA attributes include roles (like "button" or "dialog"), states (like "aria-checked"), and properties (like aria-label).
    • In total, there are about 50-60 standardized ones. Examples include "aria-labelledby", "aria-hidden", and "role=navigation".

Processes

  • Adding accessibility reviews (prior to published content)
    • Implement by setting up checkpoints & tracking sessions
  • Usage of scanning tools and manual fixes across each content type

Roles and Departments

Roles and departments vary based on the organization. They are typically:

  • Content Editors and faculty focus on document and media accessibility (like alt text and captions).
  • IT often ensure systems (like Content Management Systems) support accessibility features.
  • Software Engineers handle the code-level compliance (such as ARIA attributes).
  • Digital Accessibility Speciailists may bridge all these, guiding testing and remediation.

For presenting information, you can segment by role—give editors practical guides on media, engineers technical criteria, and IT and specialists broader standards. Each team plays its part.

See mspcc.org for more information.

Quick Mental Modal

  • HTML = built-in accessibility
  • ARIA = patch for custom/dynamic UI

Additional AA Considerations Beyond Code

  • Flashing Content Avoid content that flashes more than three times per second to prevent seizures
  • Language of Page Identify the language of the page so assitive tech can adjust accordingly
  • Error Prevention For important data (i.e., legal, financial), users should have the chance to review and correct errors
  • Text Spacing Users should be able to adjust line spacing without losing content or functionality

    To allow text spacing adjustments, you ensure your site is compatible with user stylesheets or browser controls. Avoid fixed height containers or text that can't reflow when line height or spacing is increased.

Usage of Visual Cues

  • Icons ("Note" like a lightbulb)
  • Inline badges
    • Example: "Warning" icon (like a triangle)—#FF8C00 (dark amber)
    • Example: "Key info" icon (like a star)—#0050A0 (deep blue)
    • Example: "Success"—#008000 (rich green)
  • Use accessible SVG icons
    • Font Awesome (free resource with icons like note, alert, and info symbols)
    • Free CDN (link in HTML)
    • Add Icons directly with classes
    • For accessible emphasis colors, use strong contrast
    • Use recognizable cues
  • Ensure proper color contrast

Multi-Modal Content

You can offer options:

  • Summarize key text and add a video tutorial or audio file.
  • You can use tools like text-to-speech APIs to offer audio
    • (i.e., Amazon Polly, Google Cloud Text-to-Speech)
  • Structure each page with the main content, then offer a "Watch", "Listen", or "Read More" option
    • That way, users can pick their preferred medium
    • Consistency & user choice are key
