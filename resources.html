Examples (to become Glossary)

  • Glossary
  • Content Navigation
    The primary menu, links, & interactive elements appear in the same order & location across pages. For instance, if you have a top navigation bar, it shouldn't jump to the side on another page. The labels & order of items (e.g., "Home") should remain predictable. This helps users, especially those with cognitive or memory impairments, navigate easily. In practice, design your navigation components to be shared across templates, follow a logical order, & don't shift layouts between pages.
    Message Pop-Ups
    For toast or pop-up messages, ensure they are perceiable by all users. First, they should be announced to screen readers, using ARIA live regions, so that assistive tech picks them up immediately. Second, give users enough time to read them—either allow them to dismiss the message manually or keep it visible long enough to comprehend. Finally, ensure toast notifications don't trap focus—users should be able to navigate or ignore them if they prefer

    Fixing accessibility issues
  • Use native HTML first. Only use ARIA when you have to.
    • Example (bad): <div role="button">Click</div>
    • Example (better): <button>Click</button>
  • Look into accessiBe (accessibility platform) and how it can help with accessibility
  • Save presets by role on this site
  • For emphasis, avoid underlining unless it's a link. Use bold or font-weight for emphasis, or color with sufficient contrast.

Framework/Template Types/Guidelines

  • Project Management Templates
  • Accessibility Audit Reports
  • Presented information across sprint (audit & remediate)
  • Changing vs. instructing others how to incorporate accessibility levels
  • Accessibility checks should be integrated into the sprint cycle from the start:
    • during planning (define accessibility criteria)
    • during development (engineers implement & check)
    • during test (audits before release)
  • As for templates or frameworks:
    • for updates, consider using a standard project status report template
    • for accessibility, W.C.A.G. checklists or audit report templates are key
    • for sprint documentation, consider an accessibility user story template
      • e.g., "As a user with [disability], I need [feature] so that [outcome]."
    • for resources, look at W.3.G.'s W.C.A.G. Quick Reference, agile accessibility guides, & your organization's chosen project management tools
